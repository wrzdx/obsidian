Пользователю важны красивые имена и папочки. Разработчику ОС важно: как хранить, как управлять свободным местом и как не убить диск.

## 4.3.1 Компоновка файловой системы (File-System Layout)

Файловые системы живут на дисках, разбитых на разделы (partitions). Структура зависит от того, насколько древнее у тебя железо.

### Old School: MBR (Master Boot Record)
Используется на старых системах.
*   **Сектор 0 (MBR):** Содержит код для загрузки и таблицу разделов (адреса начала и конца).
*   **Boot Process:** BIOS грузит MBR $\rightarrow$ MBR ищет активный раздел $\rightarrow$ читает первый блок раздела (**Boot block**) $\rightarrow$ загружает ОС.
*   **Структура раздела:**
    1.  **Boot block:** Есть в начале каждого раздела (даже если там нет ОС, для единообразия).
    2.  **Superblock:** Ключевые параметры ФС (magic number, количество блоков, размер). Загружается в RAM при старте.
    3.  **Free space mgmt:** Битовая карта или список свободных блоков.
    4.  **I-nodes:** Массив структур с информацией о файлах.
    5.  **Root dir:** Корневой каталог.
    6.  **Files & Directories:** Остальное место.


### New School: UEFI (Unified Extensible Firmware Interface)
Создан Intel, чтобы обойти лимиты MBR (адресация только до 2 ТБ) и убрать зависимость от реального режима x86. Поддерживает диски до $8 \text{ ZiB}$.

*   **GPT (GUID Partition Table):** Таблица разделов, использует глобальные уникальные идентификаторы. Копия GPT хранится в конце диска для надежности.
*   **Процесс:** UEFI ищет таблицу разделов не в 0-м секторе (там "заглушка" для совместимости), а во втором блоке.
*   **ESP (EFI System Partition):** UEFI умеет читать FAT-файловые системы. Загрузчик лежит там в виде обычного файла (формат .PE, как .exe в Windows). Прошивка по сути сама является мини-ОС.

## 4.3.2 Реализация файлов (Implementing Files)

Главный вопрос: как сопоставить файл с блоками на диске?

### 1. Непрерывное выделение (Contiguous Allocation)
Файл хранится как единый кусок блоков подряд.
*   **Пример:** Файл 50 КБ на блоках по 1 КБ займет 50 блоков подряд.
*   **Плюсы:**
    *   Простота: нужно хранить только адрес начала и длину.
    *   Скорость: всего 1 позиционирование головки (seek) для чтения всего файла.
*   **Минусы:**
    *   **Фрагментация:** При удалении файлов образуются "дыры". Новый файл может не влезть в дыру, даже если суммарно места хватает.
    *   Нужно знать размер файла заранее (невозможно для растущих логов).

*(Здесь вставь Рис. 4-12: Contiguous allocation)*

### 2. Связный список (Linked-List Allocation)
Файл — это связный список блоков. Первые байты блока — указатель на следующий.
*   **Плюсы:** Нет фрагментации, используется любой свободный блок.
*   **Минусы:**
    *   Медленный произвольный доступ (Random Access). Чтобы прочитать блок $N$, нужно прочитать $N-1$ предыдущих.
    *   Размер данных в блоке не степень двойки (из-за указателя), что неудобно для программ.


### 3. Связный список с таблицей в памяти (FAT)
Указатели вынесены из блоков диска в отдельную таблицу в оперативной памяти — **FAT (File Allocation Table)**.
*   **Принцип:** Ячейка таблицы соответствует блоку диска. Цепочка строится в памяти.
*   **Плюсы:** Быстрый random access (цепочка в RAM), блок диска полностью под данные.
*   **Минусы:** Таблица должна быть в памяти *целиком*.
    *   Для диска 1 ТБ и блока 1 КБ нужно $10^9$ записей. При записи 4 байта — это ~4 ГБ RAM просто под таблицу. Не масштабируется.


### 4. I-nodes (Index-nodes)
У каждого файла есть структура **i-node**, где перечислены атрибуты и адреса его блоков.
*   **Плюсы:** В памяти нужно держать i-nodes только *открытых* файлов. Экономия RAM колоссальная по сравнению с FAT.
*   **Проблема больших файлов:** Место в i-node ограничено.
    *   Решение: Последний адрес указывает не на данные, а на блок, полный адресов (**Indirect block**). Возможны двойные и тройные косвенные блоки. Используется в UNIX (и похожее в NTFS).

## 4.3.3 Реализация каталогов (Implementing Directories)

Задача каталога — перевести ASCII-имя файла в информацию для поиска данных (адрес блока или номер i-node).

### Хранение атрибутов
1.  **В каталоге:** Запись содержит имя + все атрибуты + адреса диска.
2.  **В i-node:** Каталог хранит только `Имя файла` + `Номер i-node`. Атрибуты лежат в i-node. (Классика UNIX).

### Длинные имена файлов
Раньше имена были короткими (8.3 в MS-DOS). Сейчас — до 255 и более.
*   **Вариант 1:** Отводить 255 байт на запись. Расточительно.
*   **Вариант 2 (Inline):** Запись переменной длины (Длина записи + Атрибуты + Имя).
    *   Проблема: При удалении образуются дыры разного размера.
    *   Проблема: Имя может "размазаться" между страницами памяти, вызывая page fault.
*   **Вариант 3 (Heap):** Записи каталога фиксированы, а имена свалены в кучу (heap) в конце блока каталога.

### Поиск в каталоге
*   **Линейный:** Медленно для огромных папок.
*   **Хеш-таблица:** Хешируем имя $\rightarrow$ получаем вход в таблицу. Быстро, но сложнее администрировать.
*   **Кеширование:** Сохранять результаты последних поисков.

## 4.3.4 Общие файлы (Shared Files)

Когда файл должен быть в нескольких папках одновременно (например, общая либа для проектов). ФС превращается из дерева в DAG (направленный ациклический граф).

### 1. Жесткие ссылки (Hard Links)
В каталоге создается запись, указывающая на *тот же самый* i-node.
*   **Проблема:** Если владелец удалит файл, i-node и данные исчезнут. Ссылка у второго пользователя станет битой или укажет на чужой новый файл.
*   **Решение:** Счетчик ссылок в i-node.
    *   Создание ссылки: `count++`.
    *   Удаление файла: `count--`. Данные стираются только при `count == 0`.

### 2. Символические ссылки (Symbolic Links)
Создается новый файл типа `LINK`, внутри которого записан *путь* к оригинальному файлу.
*   **Плюсы:** Можно ссылаться на файлы на других машинах/сетевых дисках. Удаление оригинала безопасно (ссылка просто перестает работать).
*   **Минусы:** Медленнее (надо читать файл-ссылку, парсить путь, искать заново). Требует дополнительный i-node.

## 4.3.5 Журналируемые ФС (Log-Structured File Systems - LFS)

**Проблема:** CPU быстрые, RAM много (кеш растет), а позиционирование диска (seek) медленное. Кеш забирает на себя все чтения. Узкое место — **запись**, особенно мелкая и случайная.

**Идея LFS:**
Представить весь диск как один бесконечный лог.
1.  Все мелкие записи буферизуются в RAM.
2.  Сбрасываются на диск одним большим куском (сегментом) в конец лога.
3.  I-nodes разбросаны по логу (нужна карта i-nodes для их поиска).

**Очистка (Garbage Collection):**
Диск конечен, лог зацикливается.
*   Специальный поток **Cleaner** читает старые сегменты с "хвоста".
*   Проверяет, актуальны ли данные (не были ли они перезаписаны в "голове" лога).
*   Живые данные копирует вперед, старый сегмент объявляет свободным.
*   **Итог:** Идеально для записи, но сложное администрирование.

## 4.3.6 Журналирование (Journaling File Systems)

**Цель:** Надежность при сбоях (отключение питания).
Операции (например, удаление файла) требуют нескольких шагов:
1.  Удалить запись в каталоге.
2.  Освободить i-node.
3.  Освободить блоки.

Если питание пропадет между шагами — ФС сломана (утечка ресурсов или, хуже, один блок у двух файлов).

**Решение:** Вести **журнал намерений**.
1.  Сначала пишем в лог: "Я собираюсь удалить файл X".
2.  Выполняем операцию.
3.  Стираем запись в логе.

При перезагрузке после сбоя система читает журнал. Если там есть незавершенные дела — просто повторяет их.
*   Важно: Операции должны быть **идемпотентными** (повтор операции не ломает систему).
*   Примеры: NTFS, ext3, ext4.

## 4.3.7 Файловые системы на Flash (SSD)

SSD отличаются от HDD физикой:
*   Нет движущихся частей $\rightarrow$ random read очень быстрый.
*   **Запись медленная:** Нельзя просто перезаписать байт. Нужно стереть блок (erase block), а потом записать.
*   **Асимметрия:** Unit of I/O (Page, ~4KB) $\ll$ Unit of Erase (Block, ~256KB).
*   **Износ:** Ячейки памяти умирают после определенного числа циклов перезаписи.

**FTL (Flash Translation Layer):** "Секретный соус" внутри SSD. Прослойка, транслирующая логические адреса блоков (от ОС) в физические страницы флеша.
*   Занимается **Wear Leveling** (размазыванием износа): часто обновляемые данные перемещаются по разным физическим блокам.
*   **Write Amplification:** Изменение 1 страницы требует чтения всего блока, стирания и записи его в новое место.

**TRIM:** Команда, которой ФС говорит контроллеру SSD: "Эти блоки я удалила, они мне не нужны". Без TRIM контроллер SSD не знает, что данные мусор, и продолжает их старательно переносить при очистке мусора, убивая ресурс диска.

**LFS на Flash:** Идеология LFS (писать только вперед, не перезаписывать на месте) идеально ложится на физику SSD.
*   *Проблема блуждающего дерева (Wandering Tree):* Обновили данные $\rightarrow$ новый адрес блока $\rightarrow$ надо обновить i-node $\rightarrow$ новый адрес i-node $\rightarrow$ надо обновить карту i-nodes. Вызывает каскад записей.
*   Решается (например, в F2FS) через глобальную таблицу трансляции номеров узлов.

## 4.3.8 Виртуальные файловые системы (VFS)

UNIX-системы объединяют множество разных ФС (ext4, NTFS, сетевой NFS, флешки) в одно дерево каталогов.
Для этого используется абстракция — **VFS**.

**Как это работает:**
1.  **Upper Interface:** Стандартные вызовы POSIX (`open`, `read`, `write`) от программ идут в VFS.
2.  **Lower Interface:** VFS определяет, к какой конкретной ФС относится запрос.
3.  У каждой конкретной ФС есть вектор функций (таблица указателей), реализующих стандартные операции (`read_block`, `get_inode` и т.д.).
4.  **Структуры данных:**
    *   **Superblock:** описание ФС.
    *   **V-node:** абстрактный i-node (представление файла в памяти). V-node содержит указатели на функции конкретной ФС.

**Пример:**
Вызов `read(fd, buf)` $\rightarrow$ VFS берет `v-node` из дескриптора $\rightarrow$ находит указатель на функцию чтения конкретной ФС (например, драйвера NTFS) $\rightarrow$ вызывает её. VFS даже не знает, где физически лежат данные.
