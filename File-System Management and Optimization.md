Заставить ФС работать — это одно. Заставить её работать быстро и надежно — совсем другое.

## 4.4.1 Управление дисковым пространством (Disk-Space Management)

Главный вопрос: как хранить файл размером $n$ байт?
Почти все современные системы разбивают файлы на блоки фиксированного размера (не обязательно смежные).

### Размер блока (Block Size)
Это классический компромисс (trade-off) между скоростью и эффективностью использования места.
*   **Большие блоки:**
    *   *Плюс:* Высокая скорость чтения (меньше операций `seek`).
    *   *Минус:* Внутренняя фрагментация. Файл в 1 байт займет весь блок (например, 4 КБ).
*   **Маленькие блоки:**
    *   *Плюс:* Экономия места (меньше "хвостов").
    *   *Минус:* Много времени уходит на поиск дорожек и вращение диска, так как файл размазан по многим блокам.

Время чтения блока данных $k$ байт можно выразить формулой:
$$ T_{access} = T_{seek} + T_{rotational} + \frac{k}{Rate_{transfer}} $$
Где $T_{seek}$ — время поиска, $T_{rotational}$ — задержка вращения.
Для HDD исторически оптимальным размером считался **4 КБ**. Но так как диски становятся огромными (ТБ и ПБ), сейчас имеет смысл увеличивать размер блока, жертвуя местом ради скорости.


### Учет свободных блоков (Keeping Track of Free Blocks)
Два основных метода:

1.  **Связный список (Linked List):**
    *   Храним список свободных блоков в... свободных блоках.
    *   Один блок хранит сотни адресов других свободных блоков.
    *   *Плюс:* Почти не тратит лишнего места (использует "пустоту").
    *   *Минус:* При сильной фрагментации список разбросан по диску.
2.  **Битовая карта (Bitmap):**
    *   $n$ блоков диска = $n$ бит в карте. 1 — свободно, 0 — занято.
    *   *Плюс:* Компактно, легко искать смежные блоки (группы нулей/единиц).
    *   *Минус:* Требует память для хранения самой карты.


**Оптимизация списка:** Хранить указательный блок в памяти наполовину полным. Это позволяет и выделять, и освобождать блоки без постоянной записи указательного блока на диск (избегаем лишнего I/O при создании временных файлов).

### Дисковые квоты (Disk Quotas)
Чтобы пользователи не забили весь диск котиками, админ ставит квоты.
*   **Soft limit:** Предупреждение, но работать можно (до логаута).
*   **Hard limit:** Жесткий отказ в записи.
Реализуется через таблицу открытых файлов, ссылающуюся на таблицу квот пользователя.

## 4.4.2 Резервное копирование (File-System Backups)
Бэкапы спасают от двух бед:
1.  **Disaster:** Сгорел диск, потоп, украли сервер.
2.  **Stupidity:** Юзер случайно удалил файл (самая частая причина).

### Стратегии дампа (Dumping strategies)
*   **Физический дамп:** Тупо копируем блоки с 0 по $N$.
    *   *Плюсы:* Прост, быстр.
    *   *Минусы:* Копирует пустые блоки и мусор; нельзя пропустить ненужные папки; нельзя сделать инкрементальный бэкап отдельных файлов.
*   **Логический дамп:** Рекурсивный обход дерева каталогов.
    *   Копируются только измененные файлы (по дате).
    *   Позволяет восстанавливать конкретные файлы.
    *   Самый популярный метод в UNIX.

### Алгоритм логического дампа (UNIX-style)
1.  **Фаза 1:** Проход по дереву, помечаем (в битовой карте) все измененные файлы и *все* каталоги.
2.  **Фаза 2:** Снимаем метки с каталогов, внутри которых нет измененных файлов. (Оставляем только путь к измененным файлам).
3.  **Фаза 3:** Дампим помеченные каталоги.
4.  **Фаза 4:** Дампим помеченные файлы.

**Проблемы:**
*   Бэкап "живой" системы опасен (файлы меняются в процессе). Нужны мгновенные снимки (snapshots).
*   Нельзя бэкапить специальные файлы (вроде `/dev/zero`), дыры в файлах (sparse files) нужно обрабатывать корректно, чтобы не раздуть архив.

## 4.4.3 Согласованность файловой системы (File-System Consistency)

Если питание вырубили, пока кеш не сброшен на диск, ФС может стать несогласованной.
Утилиты типа `fsck` (UNIX) или `sfc` (Windows) проверяют целостность после сбоя.

### Проверка блоков
Строятся две таблицы счетчиков для каждого блока: "Сколько раз блок встретился в файлах" и "Сколько раз в списке свободных".
*   **Норма:** Блок либо 1 раз в файлах, либо 1 раз в свободных.
*   **Ошибка 1 (Missing block):** Блок нигде. Решение: добавить в список свободных.
*   **Ошибка 2 (Duplicate free):** Блок дважды в списке свободных. Решение: пересобрать список.
*   **Ошибка 3 (Duplicate data block):** Один блок принадлежит двум файлам. Критично! Решение: выделить новый блок, скопировать данные, раздать каждому файлу свою копию.


### Проверка файлов (Link counts)
Сверка счетчика ссылок в i-node с реальным количеством ссылок из каталогов.
*   Если счетчик в i-node выше реальности — файл "вечный", место не освободится.
*   Если ниже — при удалении одной ссылки файл удалится, а вторая ссылка станет битой (укажет на мусор). Решение: принудительно выставить верный счетчик.

## 4.4.4 Производительность (File-System Performance)

Диск — это медленно. Память — быстро. Разница в миллион раз. Нужно оптимизировать.

### 1. Кэширование (Block Cache / Buffer Cache)
Блоки хранятся в RAM. Проверяем хеш-таблицу: есть блок в памяти? Берем. Нет? Читаем с диска.
*   Алгоритм вытеснения: обычно LRU (Least Recently Used), но с модификациями.
    *   Критические блоки (i-nodes, метаданные) пишутся на диск сразу (write-through) или очень быстро, чтобы не сломать ФС при сбое.
    *   В UNIX системный вызов `sync` (и демон `update`) сбрасывает грязные буферы каждые 30 сек.
*   **Unified Cache:** В современных ОС Buffer Cache (блоки диска) и Page Cache (страницы памяти) объединены, чтобы не хранить данные дважды.

*(Здесь вставь Рис. 4-30: The buffer cache data structures)*

### 2. Упреждающее чтение (Block Read Ahead)
Если файл читают последовательно (блок $k$), система заранее грузит блок $k+1$.
Для случайного доступа (random access) это вредно, поэтому система следит за паттерном доступа.

### 3. Сокращение движения головки
*   **Кластеризация:** Размещать блоки одного файла рядом (в одном цилиндре).
*   **Группы цилиндров (Cylinder groups):** Разбрасывать i-nodes по диску (рядом с их данными), а не кучковать их всех в начале диска (как было в ранних UNIX). Это резко сокращает `seek time`.

## 4.4.5 - 4.4.7 Дополнительные фичи

### Дефрагментация
*   **HDD:** Файлы разбросаны $\rightarrow$ головка скачет $\rightarrow$ тормоза. Дефрагментация собирает файлы в кучу. Полезно.
*   **SSD:** Дефрагментация **вредна**. Скорость доступа к любому блоку одинакова (seek = 0), а лишняя перезапись убивает ресурс ячеек памяти.

### Сжатие и дедупликация
*   **Сжатие:** ФС (ZFS, NTFS, Btrfs) может жать данные на лету.
*   **Дедупликация:** Если 100 юзеров сохранили один и тот же файл, ФС хранит одну копию, а остальным дает ссылки. Работает через хеширование чанков (chunks) данных. Бывает inline (при записи) или post-process (фоном).

### Безопасное удаление и шифрование
*   **Удаление:** `rm` просто стирает запись в каталоге. Данные лежат на диске, пока их не перезапишут. Для секретности нужен "шреддинг" (многократная перезапись мусором).
*   **Шифрование (Full Disk Encryption):**
    *   Software (BitLocker в Windows использует AES).
    *   Hardware (SED — Self-Encrypting Drives). Диск шифрует сам себя, но часто в них находят дыры.
    *   Без ключа данные — просто цифровой шум.
