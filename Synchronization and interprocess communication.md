**Необходимость:** Процессам (и потокам) часто требуется обмениваться информацией и синхронизировать свою работу.

**Три ключевые проблемы IPC:**

1.  **Передача информации:** Как один процесс (или поток) может передать данные другому.
2.  **Взаимное исключение:** Как гарантировать, что два или более процесса/потока не будут мешать друг другу при доступе к общим ресурсам (например, два потока, пытающиеся забронировать одно место на рейс).
3.  **Правильная последовательность:** Как обеспечить корректный порядок выполнения при наличии зависимостей (например, процесс B должен ждать, пока процесс A произведет данные, прежде чем начать их печатать).

**Важное замечание:** Проблемы взаимного исключения и правильной последовательности актуальны как для процессов (особенно при общей памяти), так и для потоков. Передача информации для потоков проще, так как они по своей природе делят общее адресное пространство.

## 2.4.1 Состояния Гонки (Race Conditions)

**Контекст:** Процессы, работающие совместно, могут использовать общую память (в ОЗУ или файл), к которой каждый может читать и писать.

**Пример: Диспетчер печати (Print Spooler):**
*   Процесс, желающий напечатать файл, записывает его имя в специальный каталог диспетчера.
*   Демон принтера периодически проверяет этот каталог, печатает файлы и удаляет их имена.
*   Каталог имеет слоты (0, 1, 2,...), каждый для имени файла.
*   Две общие переменные (доступные всем процессам):
    *   `out`: Указывает на следующий файл для печати.
    *   `in`: Указывает на следующий свободный слот в каталоге.

**Сценарий "состояния гонки":**
Представим, что слоты 0-3 пусты, 4-6 заняты. `out = 4`, `in = 7`.
*   **Процесс A** решает поставить файл в очередь. Он **читает `in` (получает 7)** и сохраняет его в локальной переменной `next_free_slot`.
*   Происходит **прерывание по таймеру**, и CPU переключается на **Процесс B**.
*   **Процесс B** также решает поставить файл в очередь. Он **читает `in` (также получает 7)** и сохраняет его в своей локальной переменной `next_free_slot`.
    *   *На этом этапе оба процесса считают, что следующий свободный слот — 7.*
*   **Процесс B** продолжает выполнение. Он записывает имя своего файла в **слот 7** и **обновляет `in` на 8**. Затем он уходит выполнять другие дела.
*   **Процесс A** снова запускается с того места, где остановился. Он смотрит на свою `next_free_slot` (которая все еще 7) и записывает имя **своего файла в слот 7**, **стирая имя, только что записанное Процессом B**. Затем он вычисляет `next_free_slot + 1` (получает 8) и **устанавливает `in` в 8**.

**Результат:**
*   Каталог диспетчера остается внутренне согласованным (`in = 8`).
*   Демон принтера не заметит ошибки.
*   Файл Процесса B **никогда не будет напечатан**, поскольку его имя было стёрто. Пользователь B не получит распечатку.

**Что такое "состояние гонки" (Race Condition):**
Это ситуации, когда два или более процесса (или потока) одновременно читают или записывают общие данные, и **конечный результат зависит от точной последовательности и времени выполнения операций**.
*   **Отладка:** Крайне затруднительна, так как большинство тестовых запусков могут проходить нормально, и ошибка проявляется редко и непредсказуемо.
*   **Актуальность:** С ростом числа ядер CPU и параллелизма состояния гонки становятся все более распространенными.


## 2.4.2 Критические Регионы

**Проблема:** Как избежать состояний гонки?
**Решение:** Запретить одновременный доступ более чем одному процессу (или потоку) к общим данным (общая память, файлы). Это называется **взаимным исключением (mutual exclusion)**.

**Критический регион (Critical Region/Section):**
*   Это часть программы, где происходит доступ к общим ресурсам (разделяемой памяти, файлам).
*   Цель: гарантировать, что ни в какой момент времени **не более одного процесса (или потока) находится в своем критическом регионе**.

**Четыре условия для хорошего решения проблемы критических регионов:**

1.  **Взаимное исключение:** Никакие два процесса не могут одновременно находиться в своих критических регионах.
2.  **Независимость от скорости:** Не должно делаться никаких предположений о скорости выполнения процессов или количестве CPU.
3.  **Свобода от блокировки:** Процесс, выполняющийся вне своего критического региона, не должен блокировать любой другой процесс.
4.  **Отсутствие голодания (No Starvation):** Ни один процесс не должен ждать вечно, чтобы войти в свой критический регион.

**Иллюстрация поведения (см. рис. 2-22):**

*   Процесс A входит в свой критический регион в момент T1.
*   В момент T2 Процесс B пытается войти в свой критический регион, но ему это не удается (поскольку Процесс A уже находится там). Процесс B временно приостанавливается.
*   В момент T3 Процесс A выходит из своего критического региона, после чего Процесс B немедленно входит в свой критический регион.
*   В момент T4 Процесс B выходит, и система возвращается в исходное состояние, когда ни один процесс не находится в критическом регионе.

## Взаимное Исключение с Активным Ожиданием (Busy Waiting)

В этом разделе рассматриваются различные подходы к достижению взаимного исключения, при которых процессы, ожидающие входа в критический регион, активно проверяют условие в цикле (busy waiting).

**1. Отключение прерываний (Disabling Interrupts)**

*   **Идея:** На однопроцессорной системе процесс отключает все прерывания при входе в критический регион и включает их при выходе. Это предотвращает переключение CPU на другой процесс.
*   **Преимущества:** Простота реализации на однопроцессорных системах.
*   **Недостатки:**
    *   **Опасно:** Предоставление пользовательским процессам права отключать прерывания опасно (может привести к зависанию системы).
    *   **Не работает в многопроцессорных системах:** Отключение прерываний затрагивает только тот CPU, который выполнил инструкцию. Другие CPU могут продолжать работать и получать доступ к общей памяти.
*   **Применимость:** Часто используется внутри самого ядра ОС для обновления критических структур данных на короткий промежуток времени, но не подходит как общий механизм для пользовательских процессов. В многоядерных системах даже для ядра требуются более сложные схемы.

**2. Переменная-флаг (Lock Variable)**

*   **Идея:** Используется одна общая переменная `lock`, изначально 0. Процесс, желающий войти в критический регион, проверяет `lock`. Если 0, устанавливает ее в 1 и входит. Если 1, ждет, пока станет 0.
*   **Недостаток:** **Подвержен состоянию гонки.**
    *   Процесс A читает `lock` (видит 0).
    *   До того, как A успеет установить `lock` в 1, происходит переключение на Процесс B.
    *   Процесс B читает `lock` (также видит 0), устанавливает ее в 1 и входит в критический регион.
    *   Процесс A возобновляет работу, устанавливает `lock` в 1 и тоже входит в критический регион.
    *   В результате два процесса находятся в критическом регионе одновременно.

**3. Строгое чередование (Strict Alternation)**

*   **Идея:** Используется общая целочисленная переменная `turn`, изначально 0, которая указывает, чей сейчас черед входить в критический регион.
    *   Процесс 0: `while (turn != 0) {}` (активное ожидание), затем `critical_region()`, затем `turn = 1`.
    *   Процесс 1: `while (turn != 1) {}` (активное ожидание), затем `critical_region()`, затем `turn = 0`.
*   **Преимущества:** Гарантирует взаимное исключение.
*   **Недостатки:**
    *   **Нарушает условие 3 (свобода от блокировки):** Процесс, находящийся вне критического региона, может блокировать другой процесс. Если Процесс 0 быстро завершает критический регион и некритический регион, а Процесс 1 очень медленно работает в своем некритическом регионе, Процесс 0 будет заблокирован, ожидая `turn = 0`, хотя никто не находится в критическом регионе. Это является **активным ожиданием (busy waiting)** и тратит CPU.
    *   Требует строгого чередования, что неэффективно, если процессы имеют разную скорость.
*   **Активное ожидание (Busy Waiting):** Постоянная проверка переменной в цикле. Используется только если ожидание, предположительно, будет очень коротким. Блокировка, использующая активное ожидание, называется **спин-блокировкой (spin lock)**.

**4. Решение Питерсона (Peterson’s Solution) — 1981 год**

*   **Идея:** Комбинирует идеи переменных-флагов и чередования. Является чисто программным решением, не требующим строгого чередования.
*   **Переменные:**
    *   `turn`: Чей черед.
    *   `interested[N]`: Массив флагов, показывающий, заинтересован ли процесс во входе в критический регион.
*   **Функция `enter_region(int process)`:**
    1.  `other = 1 - process;`
    2.  `interested[process] = TRUE;` (Показывает заинтересованность)
    3.  `turn = process;` (Устанавливает, что сейчас очередь текущего процесса)
    4.  `while (turn == process && interested[other] == TRUE);` (Активное ожидание: ждать, пока не будет очередь другого ИЛИ другой процесс не будет заинтересован).
*   **Функция `leave_region(int process)`:**
    1.  `interested[process] = FALSE;` (Показывает отсутствие заинтересованности)
*   **Как работает:**
    *   Если один процесс вызывает `enter_region`, он устанавливает свою заинтересованность и `turn` в свой номер. Так как другой не заинтересован, цикл `while` сразу завершается, и процесс входит в критический регион.
    *   Если оба процесса вызывают `enter_region` почти одновременно, оба установят `turn` в свой номер. Значение `turn` будет последним записанным. Например, если Процесс 1 записал последним, `turn = 1`.
        *   Процесс 0 увидит `turn == 1` и `interested[1] == TRUE`, поэтому будет ждать в цикле `while`.
        *   Процесс 1 увидит `turn == 1` и `interested[0] == TRUE`, но так как `turn == process` (т.е. `turn == 1`), он будет ждать в цикле `while`.
        *   **Исправление:** (Подразумевается, что в оригинальном тексте есть опечатка или упрощение). В корректном решении Питерсона, если `turn` установлен в номер текущего процесса, а другой процесс заинтересован, текущий процесс должен ждать. Если `turn` установлен в номер другого процесса, а другой процесс заинтересован, текущий процесс должен ждать. Суть в том, что `turn` используется для разрешения конфликтов, если оба заинтересованы.
        *   **Корректная логика:** Если оба хотят войти, `turn` определяет, кто получит приоритет. Например, если `turn = 0`, Процесс 0 заходит, а Процесс 1 ждет. Если `turn = 1`, Процесс 1 заходит, а Процесс 0 ждет.

**5. Инструкция TSL (Test and Set Lock)**

*   **Идея:** Аппаратная поддержка. Некоторые CPU имеют атомарную инструкцию `TSL RX, LOCK`. Она читает содержимое слова памяти `LOCK` в регистр `RX`, затем записывает ненулевое значение в `LOCK`. **Эти две операции гарантированно неделимы (атомарны).** CPU блокирует шину памяти, чтобы другие CPU не могли получить доступ к `LOCK` во время выполнения `TSL`.
*   **Отличие от отключения прерываний:** Блокировка шины предотвращает интерференцию от **других CPU**, тогда как отключение прерываний действует только на текущий CPU.
*   **Использование:**
    *   Общая переменная `lock`, изначально 0.
    *   **Вход в критический регион (`enter_region`):**
        1.  `TSL REGISTER, LOCK` (читает `LOCK` в `REGISTER`, устанавливает `LOCK` в 1).
        2.  `CMP REGISTER, #0` (сравнивает старое значение `LOCK` с 0).
        3.  `JNE enter_region` (если `REGISTER` не 0, значит, `LOCK` был установлен, и другой процесс уже в критическом регионе, поэтому цикл продолжается).
        4.  `RET` (если `REGISTER` был 0, `LOCK` установлен в 1, и процесс входит в критический регион).
    *   **Выход из критического региона (`leave_region`):**
        1.  `MOVE LOCK, #0` (устанавливает `LOCK` обратно в 0 обычной инструкцией).
        2.  `RET`
*   **Преимущества:** Атомарно гарантирует взаимное исключение.
*   **Недостатки:** Требует **активного ожидания**.
*   **Требование к сотрудничеству:** Как и во всех решениях с критическими регионами, это работает только при условии, что процессы добросовестно вызывают `enter_region` и `leave_region`.

**6. Инструкция XCHG (Exchange)**

*   **Идея:** Аппаратная поддержка, аналогичная `TSL`. Инструкция `XCHG REGISTER, LOCK` атомарно обменивает содержимое регистра и слова памяти.
*   **Использование:** Аналогично `TSL`, с тем же кодом для `enter_region` и `leave_region`. Широко используется в процессорах Intel x86.

## 2.4.4 Sleep and Wakeup (Сон и Пробуждение)

#### Проблема активного ожидания (Busy Waiting)
Предыдущие решения (алгоритм Петерсона, TSL) рабочие, но имеют жирный минус: **активное ожидание**.
*   Если вход в критическую секцию запрещен, процесс крутится в бесконечном цикле, проверяя условие.
*   Это впустую жжет такты CPU.

**Инверсия приоритетов (Priority Inversion):**
Это опасный побочный эффект активного ожидания.
1.  Есть процесс **L** (Low priority) и **H** (High priority).
2.  **L** входит в критическую секцию.
3.  Просыпается **H** (например, завершился I/O) и вытесняет **L** (так как приоритет выше).
4.  **H** пытается войти в секцию, занятую **L**, и начинает "крутиться" (busy wait).
5.  Так как **H** занял процессор, **L** не получает времени, чтобы закончить работу и выйти из секции.
6.  **Итог:** **H** вечно ждет выхода **L**, а **L** не может работать из-за **H**. Система виснет.

### Блокирующие примитивы
Чтобы не жечь CPU, используются системные вызовы для блокировки:
*   **`sleep()`** — процесс блокируется (засыпает) и отдает управление планировщику, пока его не разбудят.
*   **`wakeup(process)`** — будит указанный процесс (переводит в состояние "готов").

## Проблема Производителя и Потребителя (Bounded-buffer problem)
Классическая задача для демонстрации синхронизации.

**Дано:**
*   Буфер фиксированного размера на $N$ слотов.
*   **Производитель (Producer):** кладет данные в буфер.
*   **Потребитель (Consumer):** забирает данные.
*   Переменная `count`: следит за количеством элементов.

**Логика работы:**
1.  Если буфер полон, Производитель вызывает `sleep()`.
2.  Если буфер пуст, Потребитель вызывает `sleep()`.
3.  Когда Производитель кладет первый элемент (было 0 -> стало 1), он будит Потребителя (`wakeup`).
4.  Когда Потребитель освобождает место (было $N$ -> стало $N-1$), он будит Производителя.

````c
#define N 100
int count = 0;

/* number of slots in the buffer */
/* number of items in the buffer */
void producer(void)
{
	int item;
	while (TRUE) {
	item = produce item( );
	if (count == N) sleep( );
	inser t item(item);
	count = count + 1;
	if (count == 1) wakeup(consumer);
}
/* repeat forever */
/* generate next item */
/* if buffer is full, go to sleep */
/* put item in buffer */
/* increment count of items in buffer */
/* was buffer empty? */
}
void consumer(void)
{
	int item;
	while (TRUE) {
	if (count == 0) sleep( );
	item = remove item( );
	count = count < 1;
	if (count == N < 1) wakeup(producer);
	consume item(item);
}
/* repeat forever */
/* if buffer is empty, got to sleep */
/* take item from buffer */
/* decrement count of items in buffer */
/* was buffer full? */
/* print item */
}
````

### Fatal Race Condition (Фатальная гонка)
Код на рис. 2-27 содержит критическую уязвимость из-за того, что доступ к `count` не защищен. Проблема возникает из-за **потерянного сигнала пробуждения (Lost Wakeup)**.

**Сценарий краша:**
1.  Буфер пуст (`count = 0`).
2.  **Потребитель** читает `count`, видит 0.
3.  *В этот момент планировщик переключает контекст на Производителя.* Потребитель еще **не успел** уснуть!
4.  **Производитель** вставляет элемент, делает `count = 1`.
5.  Производитель думает: "Было 0, значит, Потребитель спит". Вызывает `wakeup(Consumer)`.
6.  **Сигнал wakeup теряется**, так как Потребитель фактически еще не спит (он только собирался).
7.  Управление возвращается **Потребителю**. Он продолжает с той точки, где остановился: вызывает `sleep()`, так как помнит, что `count` был 0.
8.  Потребитель спит вечно.
9.  Производитель заполняет буфер до $N$ и тоже засыпает.
10. **Deadlock:** Оба процесса спят вечно.

#### Попытка фикса: Бит ожидания (Wakeup Waiting Bit)
Можно добавить специальный бит, который запоминает сигнал пробуждения:
*   Если `wakeup` посылается бодрствующему процессу, этот бит устанавливается в 1.
*   Когда процесс пытается сделать `sleep`, он проверяет бит: если 1 — бит сбрасывается, а процесс **не ложится спать**.

**Почему это плохое решение:**
Работает для двух процессов, но не масштабируется. Если процессов много, одного бита не хватит. Наращивание битов — это "костыль", а не решение фундаментальной проблемы синхронизации.
