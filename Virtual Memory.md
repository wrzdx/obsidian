**Проблема:** Программы стали жирнее, чем оперативная память (RAM).
**Старое решение:** Overlays (Оверлеи). Программист вручную бил программу на куски и загружал их по очереди. Сложно и больно.
**Новое решение:** Виртуальная память (1961).
*   Каждая программа имеет свое **виртуальное адресное пространство**, разбитое на блоки — **Страницы (Pages)**.
*   В физической памяти находятся только *нужные сейчас* страницы. Остальные лежат на диске.
*   Когда программа обращается к адресу, которого нет в RAM -> ОС подгружает его автоматически.

## 3.3.1 Paging (Страничная организация)
*   **Virtual Address (Виртуальный адрес):** То, что использует программа (MOV REG, 1000).
*   **Physical Address (Физический адрес):** Реальный адрес в чипе RAM.
*   **MMU (Memory Management Unit):** Железка, которая на лету переводит Виртуальный адрес -> Физический.

**Как это работает:**
1.  Виртуальная память делится на **Pages** (Страницы, обычно 4 КБ).
2.  Физическая память делится на **Page Frames** (Фреймы, тоже 4 КБ).
3.  **Page Table (Таблица страниц):** Словарь, где написано: "Виртуальная страница 5 лежит в Физическом фрейме 3".
4.  Если фрейма нет в таблице (Present bit = 0) -> Происходит **Page Fault** (Страничное прерывание). ОС ловит его, читает данные с диска в свободный фрейм и обновляет таблицу.

![800](Pasted%20image%2020251119233629.png)

**Структура адреса:**
Адрес делится на две части: `[Номер страницы] + [Смещение (Offset)]`.
*   Смещение не меняется (оно внутри страницы).
*   MMU заменяет номер виртуальной страницы на номер физического фрейма.

## 3.3.2 Page Tables (Таблицы страниц)
Таблица — это функция: `Virtual Page Number -> Physical Frame Number`.
**Page Table Entry (Запись в таблице):**
*   **Page Frame Number:** Главное поле.
*   **Present/Absent bit:** 1 = в памяти, 0 = на диске (вызовет Page Fault).
*   **Protection bits:** R/W/X (чтение/запись/исполнение).
*   **Modified (Dirty) bit:** Страница была изменена? (Нужно ли сохранять на диск при выгрузке).
*   **Referenced bit:** К странице обращались? (Нужно для алгоритмов вытеснения).

## 3.3.3 Speeding Up Paging (Ускорение)
**Проблемы:**
1.  Маппинг нужен на *каждый* доступ к памяти (очень часто).
2.  Таблицы страниц огромны (для 64-бит систем).

**Решение 1: TLB (Translation Lookaside Buffer)**
*   Это **кэш** внутри MMU. Хранит последние переводы "Виртуальная -> Физическая".
*   **TLB Hit:** Адрес найден в TLB -> Перевод мгновенный (без похода в память).
*   **TLB Miss:** Адреса нет -> Идем в таблицу страниц (медленно), обновляем TLB.
*   **Soft Miss:** Страница в RAM, но нет в TLB (просто обновить TLB).
*   **Hard Miss:** Страницы нет в RAM (нужен диск, очень долго).

**Управление TLB:**
*   *Hardware:* Процессор сам лезет в таблицу при промахе (x86).
*   *Software:* Процессор кидает исключение, ОС сама ищет страницу и грузит в TLB (RISC: MIPS, SPARC).

## 3.3.4 Page Tables for Large Memories (Для большой памяти)
Если адрес 32-64 бита, простая таблица будет весить гигабайты.

**Решение 1: Multilevel Page Tables (Многоуровневые таблицы)**
*   Дерево таблиц.
*   `Top-level Table` указывает на `Second-level Table`.
*   Если кусок памяти не используется (дырка), мы не создаем для него таблицы второго уровня. Экономим память.
*   *Пример:* 32-битный x86 использовал 2 уровня (Directory -> Table). 64-битные используют 4-5 уровней.

**Решение 2: Inverted Page Tables (Инвертированные таблицы)**
*   Вместо таблицы по виртуальным адресам, делаем таблицу по **физическим фреймам**.
*   В системе всего одна таблица размером с RAM (например, 1 запись на каждые 4 КБ RAM).
*   *Плюс:* Мало места занимает.
*   *Минус:* Нельзя просто взять индекс (виртуальный адрес). Нужно искать хешем по всей таблице. Используется редко (PowerPC, Itanium), требует TLB.

## 3.4 Page Replacement Algorithms (Алгоритмы замещения страниц)
**Задача:** Выбрать жертву (страницу) для удаления из RAM при Page Fault.
**Цель:** Выбрать страницу, которая *не понадобится* в ближайшее время. Если выкинуть нужную — она сразу вернется, вызывая тормоза (Thrashing).
**Критерии:**
*   **Clean page (Чистая):** Не менялась. Можно просто стереть (копия есть на диске).
*   **Dirty page (Грязная):** Менялась. Нужно сначала записать на диск (дорогая операция).

---

### 1. The Optimal Algorithm (Оптимальный)
*   **Идея:** Выкинуть страницу, которая не понадобится **дольше всего** в будущем.
*   **Проблема:** Невозможно реализовать, так как мы не знаем будущего.
*   **Польза:** Используется как эталон для сравнения других алгоритмов (в симуляторах).

### 2. NRU (Not Recently Used)
Использует биты **R** (Referenced) и **M** (Modified).
Классы страниц (от лучшего кандидата на удаление к худшему):
0.  Not Referenced, Not Modified (Не нужна, Чистая).
1.  Not Referenced, Modified (Не нужна, Грязная — *бывает, если R сбросил таймер*).
2.  Referenced, Not Modified.
3.  Referenced, Modified.
*   **Алгоритм:** Удаляем случайную страницу из самого низкого непустого класса.
*   **Плюс:** Просто. **Минус:** Грубовато.

### 3. FIFO (First-In, First-Out)
*   **Идея:** Очередь. Удаляем самую старую загруженную страницу.
*   **Проблема:** Старая страница может быть очень нужной (например, ядро программы). FIFO тупо выкинет её.

### 4. Second Chance (Второй шанс)
*   **Идея:** Модификация FIFO. Смотрим на самую старую страницу.
    *   Если **R=0** (не использовалась) -> Выкидываем.
    *   Если **R=1** (использовалась) -> Даем "второй шанс": сбрасываем R в 0, перемещаем в конец очереди (как будто только что загрузили).
*   **Плюс:** Спасает важные старые страницы.

### 5. Clock (Часы)
*   **Идея:** Тот же Second Chance, но реализация через **циклический список** (стрелка часов). Не нужно двигать элементы списка.
*   Стрелка указывает на страницу. Если R=1 -> R=0, стрелка дальше. Если R=0 -> Выкидываем.
*   **Плюс:** Быстрее, чем Second Chance.

### 6. LRU (Least Recently Used)
*   **Идея:** Выкинуть ту, которая не использовалась **дольше всех** в прошлом. (Аппроксимация Оптимального).
*   **Реализация:** Требует железа (счетчик на каждое обращение к памяти) или сложного списка. Очень дорого.

### 7. NFU (Not Frequently Used) & Aging (Старение)
Программная эмуляция LRU.
*   **Aging:** У каждой страницы есть счетчик (например, 8 бит).
*   Каждый тик таймера: Сдвигаем счетчик вправо (`>> 1`), в старший бит записываем **R**.
*   При удалении: Берем страницу с **минимальным** счетчиком.
*   **Плюс:** Отлично работает, требует мало памяти. Один из лучших алгоритмов.

---

### 8. Working Set (Рабочий набор)
*   **Концепция:** У процесса есть набор страниц, с которыми он активно работает прямо сейчас ($w(k,t)$).
*   **Thrashing (Трэшинг):** Если памяти меньше, чем размер рабочего набора -> постоянные подкачки страниц -> тормоза.
*   **Алгоритм:**
    *   Найти страницу, которая **не входит** в рабочий набор (давно не использовалась, Age > Limit).
    *   Если все страницы в рабочем наборе — выкинуть самую старую "чистую".
*   **Плюс:** Умный. **Минус:** Сложно сканировать всю таблицу.

### 9. WSClock (Working Set Clock)
*   **Идея:** Гибрид Clock и Working Set.
*   Крутим стрелку часов.
*   Если `R=1` -> Сбрасываем R, обновляем время последнего использования.
*   Если `R=0` -> Проверяем возраст (Age).
    *   Если `Age > Tau` (старая) и страница чистая -> **Выкидываем**.
    *   Если грязная -> Ставим в очередь на запись на диск, идем дальше.
*   **Итог:** Самый популярный алгоритм в реальных системах (эффективный и простой).

## 3.5 Design Issues for Paging Systems (Проблемы проектирования)

#### 3.5.1 Local vs Global Allocation (Локальное и Глобальное распределение)
Вопрос: Когда у процесса кончается память, у кого отбирать страницу?
1.  **Local (Локальное):** Отбираем страницу у **этого же** процесса.
    *   Каждому процессу выделен фиксированный "загон".
    *   *Минус:* Неэффективно. Если процесс вырос (Working Set увеличился), он начнет тормозить (Thrashing), даже если у соседа куча свободной памяти.
2.  **Global (Глобальное):** Отбираем страницу у **любого** процесса в системе.
    *   Память — общий ресурс.
    *   *Плюс:* Работает лучше, так как память перетекает туда, где она нужнее.
    *   *Контроль:* Чтобы процесс не захватил всё, используют **PFF (Page Fault Frequency)**.
        *   Если частота ошибок высока -> Даем процессу больше страниц.
        *   Если ошибок мало -> Отбираем лишние страницы.


#### 3.5.2 Load Control (Контроль нагрузки)
Что делать, если **Thrashing** все-таки начался (сумма Working Sets > RAM)?
*   Алгоритмы замещения тут уже не помогут.
*   **Решение 1 (Жесткое):** **OOM Killer (Out Of Memory Killer)**. ОС находит "плохой" процесс (жрет много памяти, но не важный) и убивает его.
*   **Решение 2 (Мягкое):** **Swapping (Свопинг процессов)**. ОС выбирает процесс, полностью выгружает его на диск и замораживает, освобождая всю его память для других. Позже вернет обратно.

#### 3.5.3 Cleaning Policy (Политика очистки)
Нельзя ждать, пока память кончится совсем.
*   **Paging Daemon:** Фоновый процесс. Просыпается периодически.
*   Если свободных фреймов мало -> ищет жертв.
*   Если жертва "грязная" -> записывает её на диск заранее, делая "чистой".
*   *Цель:* Чтобы в момент Page Fault у нас всегда были **готовые чистые фреймы**, которые можно перезаписать мгновенно.

#### 3.5.4 Page Size (Размер страницы)
ОС может выбирать размер страницы (обычно 4 КБ, но бывает и 4 МБ).
Компромисс (Trade-off):
*   **Маленькие страницы:**
    *   (+) Меньше **Internal Fragmentation** (внутренней фрагментации). Хвост последнего файла занимает меньше места.
    *   (-) Огромные таблицы страниц.
    *   (-) Медленная загрузка (много мелких обращений к диску).
*   **Большие страницы:**
    *   (+) Маленькие таблицы страниц.
    *   (+) Экономия TLB (одна запись TLB покрывает большой кусок памяти).
    *   (-) Много памяти тратится впустую на "хвостах" (если процесс занимает 5 КБ, а страница 4 МБ, то 3.99 МБ пропадают).
*   *Оптимум:* Сейчас 4 КБ — стандарт, но для БД и тяжелых приложений используют **Huge Pages** (2 МБ/1 ГБ).

#### 3.5.5 Separate Instruction and Data Spaces (Разделение I-space и D-space)
*   В старых/маленьких системах (PDP-11) адресного пространства не хватало.
*   Решение: Сделать два пространства по 64 КБ (или 4 ГБ). Одно для Кода (Instruction), другое для Данных (Data).
*   Это удваивает доступную память.
*   Сейчас это используется в кэшах процессора (L1 I-Cache и L1 D-Cache).

#### 3.5.6 Shared Pages (Разделяемые страницы)
Если два юзера запустили одну и ту же программу (например, `vim`), нет смысла грузить код дважды.
*   Код (Text) помечается **Read-Only** и используется обоими процессами.
*   Данные (Data) у каждого свои.
*   **Copy-on-Write (Копирование при записи):** При `fork()` (создании процесса в Unix) память не копируется. Оба процесса читают одни и те же страницы. Как только кто-то пытается *записать* -> происходит Page Fault, ОС создает персональную копию этой страницы. Это экономит память и время старта.

#### 3.5.7 Shared Libraries (Разделяемые библиотеки / DLL)
Программы используют стандартные функции (`printf`, GUI).
*   **Static Linking:** Код функции вшивается в каждый EXE. Файлы огромные, память тратится зря.
*   **Dynamic Linking (DLL/.so):**
    *   Библиотека загружается в память **один раз**.
    *   Все программы мапят её страницы в свой адресный список.
    *   *Проблема:* Библиотека может попасть по разным виртуальным адресам в разных процессах.
    *   *Решение:* **Position-Independent Code (PIC)**. Код компилируется так, чтобы использовать относительные переходы (`JUMP +100`), а не абсолютные адреса.

#### 3.5.8 Mapped Files (Отображаемые файлы)
Механизм `mmap`.
*   Файл "проецируется" на память.
*   Ты работаешь с ним как с массивом байтов (`array[5] = 'a'`).
*   ОС сама подгружает куски файла (Page Faults) и сохраняет изменения на диск.
*   Это удобно для баз данных и для IPC (обмен данными между процессами через общую память).

## 3.6 Implementation Issues (Проблемы реализации)

#### 3.6.1 OS Involvement with Paging (Участие ОС)
ОС работает с памятью в 4 момента:
1.  **Process Creation (Создание):** Нужно определить размер, создать таблицу страниц, выделить место на Swap-разделе (диске).
2.  **Process Execution (Исполнение):** При переключении контекста (Context Switch) нужно сбросить MMU (загрузить новый указатель на таблицу страниц) и сбросить TLB (flush), чтобы старые адреса не мешали новому процессу.
3.  **Page Fault (Ошибка страницы):** Самое сложное. Найти адрес, найти место в RAM, выгнать кого-то (evict), прочитать с диска, откатить инструкцию назад.
4.  **Process Termination (Завершение):** Освободить память и место на диске.

#### 3.6.2 Page Fault Handling (Обработка ошибки страницы)
Что реально происходит, когда программа лезет "не туда":
1.  Железо (MMU) кидает **Trap (Ловушку)** в ядро.
2.  Сохраняются регистры (Assembly wrapper).
3.  ОС выясняет **виртуальный адрес**, который вызвал сбой (обычно из спец-регистра CPU, например CR2 в x86).
4.  Проверка: адрес валидный?
    *   Нет -> `Segmentation Fault` (убить процесс).
    *   Да (просто на диске) -> Ищем свободный фрейм.
5.  Если фреймов нет -> запускаем алгоритм замещения (выгоняем жертву).
6.  Читаем нужную страницу с диска (процесс спит/blocked).
7.  Обновляем таблицы страниц.
8.  **Instruction Backup (Откат инструкции):** Самый адский этап. Нужно вернуть процессор в состояние **ДО** начала выполнения сбойной инструкции.

#### 3.6.3 Instruction Backup (Откат инструкции)
Почему это сложно?
Пример: `MOV.L #6(A1), 2(A0)` (Команда с автоинкрементом).
*   В процессе выполнения процессор мог уже изменить регистры A1 или A0 (увеличить их).
*   Если просто перезапустить команду, она снова увеличит регистры -> данные испортятся.
*   ОС должна понять, что успел сделать процессор, и **откатить** изменения регистров программно. Это кошмар разработчиков ОС.

#### 3.6.4 Locking Pages in Memory (Блокировка/Pinning страниц)
**Проблема:**
1.  Процесс А просит диск прочитать данные в буфер (страницу X). Это делает DMA-контроллер напрямую в память.
2.  Процесс А засыпает.
3.  Начинается Page Fault у другого процесса. Алгоритм (глобальный) решает выкинуть страницу X, потому что она "давно не использовалась".
4.  **Катастрофа:** Диск (DMA) продолжает писать данные в физический кадр, который теперь принадлежит другому процессу!

**Решение:** **Pinning (Закрепление)**.
Пока идет I/O, страница помечается как "невытесняемая" (Pinned/Locked). Алгоритм замещения не имеет права её трогать.

#### 3.6.5 Backing Store (Хранилище / Swap)
Куда девать страницы, когда они не в RAM?
1.  **Выделенный Swap-раздел (Swap Partition):**
    *   Сырой кусок диска без файловой системы. Быстро, так как адресация по блокам (offset), а не по файлам.
    *   *Static Swap Area:* Каждому процессу при старте выделяется кусок на диске размером с его память. (Рис 3-28а).
    *   *Dynamic:* Память выделяется на лету.
2.  **Swap File (Файл подкачки):**
    *   Обычный большой файл в файловой системе (как в Windows `pagefile.sys`). Чуть медленнее, но гибче (легко менять размер).

**Оптимизация:** Код программы (Text segment) не нужно писать в Swap. Он и так есть в `exe` файле. При вытеснении его просто стирают, а при нужде — читают заново из `exe`.

#### 3.6.6 Separation of Policy and Mechanism (Политика и Механизм)
Принцип микроядер (например, Mach).
*   **Kernel (Ядро):** Делает только **механизм** (ловит Page Fault, управляет MMU, пересылает данные).
*   **User-space External Pager (Внешний пэйджер):** Решает **политику** (какую страницу загрузить, кого выкинуть).
*   *Плюс:* Гибкость. Можно написать свой умный менеджер памяти для базы данных.
*   *Минус:* Медленно (много переключений контекста Kernel <-> User).

**Must know для экзамена:**
1.  **Последовательность Page Fault:** (Trap -> Save state -> Find Victim -> Disk Read -> Restore Instruction).
2.  **Pinning (Locking):** Зачем нужно при DMA/IO.
3.  **Backing Store:** Разница между Swap Partition и Swap File.
4.  **Instruction Restart:** Почему это больно (автоинкременты).
