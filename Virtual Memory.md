**Проблема:** Программы стали жирнее, чем оперативная память (RAM).
**Старое решение:** Overlays (Оверлеи). Программист вручную бил программу на куски и загружал их по очереди. Сложно и больно.
**Новое решение:** Виртуальная память (1961).
*   Каждая программа имеет свое **виртуальное адресное пространство**, разбитое на блоки — **Страницы (Pages)**.
*   В физической памяти находятся только *нужные сейчас* страницы. Остальные лежат на диске.
*   Когда программа обращается к адресу, которого нет в RAM -> ОС подгружает его автоматически.

## 3.3.1 Paging (Страничная организация)
*   **Virtual Address (Виртуальный адрес):** То, что использует программа (MOV REG, 1000).
*   **Physical Address (Физический адрес):** Реальный адрес в чипе RAM.
*   **MMU (Memory Management Unit):** Железка, которая на лету переводит Виртуальный адрес -> Физический.

**Как это работает:**
1.  Виртуальная память делится на **Pages** (Страницы, обычно 4 КБ).
2.  Физическая память делится на **Page Frames** (Фреймы, тоже 4 КБ).
3.  **Page Table (Таблица страниц):** Словарь, где написано: "Виртуальная страница 5 лежит в Физическом фрейме 3".
4.  Если фрейма нет в таблице (Present bit = 0) -> Происходит **Page Fault** (Страничное прерывание). ОС ловит его, читает данные с диска в свободный фрейм и обновляет таблицу.

**Структура адреса:**
Адрес делится на две части: `[Номер страницы] + [Смещение (Offset)]`.
*   Смещение не меняется (оно внутри страницы).
*   MMU заменяет номер виртуальной страницы на номер физического фрейма.

## 3.3.2 Page Tables (Таблицы страниц)
Таблица — это функция: `Virtual Page Number -> Physical Frame Number`.
**Page Table Entry (Запись в таблице):**
*   **Page Frame Number:** Главное поле.
*   **Present/Absent bit:** 1 = в памяти, 0 = на диске (вызовет Page Fault).
*   **Protection bits:** R/W/X (чтение/запись/исполнение).
*   **Modified (Dirty) bit:** Страница была изменена? (Нужно ли сохранять на диск при выгрузке).
*   **Referenced bit:** К странице обращались? (Нужно для алгоритмов вытеснения).

## 3.3.3 Speeding Up Paging (Ускорение)
**Проблемы:**
1.  Маппинг нужен на *каждый* доступ к памяти (очень часто).
2.  Таблицы страниц огромны (для 64-бит систем).

**Решение 1: TLB (Translation Lookaside Buffer)**
*   Это **кэш** внутри MMU. Хранит последние переводы "Виртуальная -> Физическая".
*   **TLB Hit:** Адрес найден в TLB -> Перевод мгновенный (без похода в память).
*   **TLB Miss:** Адреса нет -> Идем в таблицу страниц (медленно), обновляем TLB.
*   **Soft Miss:** Страница в RAM, но нет в TLB (просто обновить TLB).
*   **Hard Miss:** Страницы нет в RAM (нужен диск, очень долго).

**Управление TLB:**
*   *Hardware:* Процессор сам лезет в таблицу при промахе (x86).
*   *Software:* Процессор кидает исключение, ОС сама ищет страницу и грузит в TLB (RISC: MIPS, SPARC).

## 3.3.4 Page Tables for Large Memories (Для большой памяти)
Если адрес 32-64 бита, простая таблица будет весить гигабайты.

**Решение 1: Multilevel Page Tables (Многоуровневые таблицы)**
*   Дерево таблиц.
*   `Top-level Table` указывает на `Second-level Table`.
*   Если кусок памяти не используется (дырка), мы не создаем для него таблицы второго уровня. Экономим память.
*   *Пример:* 32-битный x86 использовал 2 уровня (Directory -> Table). 64-битные используют 4-5 уровней.

**Решение 2: Inverted Page Tables (Инвертированные таблицы)**
*   Вместо таблицы по виртуальным адресам, делаем таблицу по **физическим фреймам**.
*   В системе всего одна таблица размером с RAM (например, 1 запись на каждые 4 КБ RAM).
*   *Плюс:* Мало места занимает.
*   *Минус:* Нельзя просто взять индекс (виртуальный адрес). Нужно искать хешем по всей таблице. Используется редко (PowerPC, Itanium), требует TLB.
