- **Назначение:** интерфейс между пользовательскими программами и ОС; позволяет работать с абстракциями (файлы, процессы) и управлять ресурсами (память, устройства).
- **Суть:** пользовательский процесс вызывает системный вызов (system call), который выполняется в **ядре ОС**. В обычном режиме (user mode) процесс не может напрямую обращаться к ресурсам ядра.
	![800](Pasted%20image%2020251117203751.png)
- **Механизм вызова:**
    
    1. Пользовательская программа готовит параметры вызова (например, файл, буфер, количество байт для `read`).
    2. Вызывается библиотечная процедура с тем же именем (`read`, `write` и т.д.).
    3. Номер системного вызова помещается в специальный регистр (например, RAX на x86-64).
    4. Выполняется **trap** (SYSCALL) — переключение в kernel mode.
    5. Ядро выполняет обработку вызова через таблицу системных вызовов.
    6. После завершения работы управление возвращается в пользовательскую программу.
    
- **Пример:** `count = read(fd, buffer, nbytes);`
    - Возвращает фактическое количество прочитанных байт.
    - В случае ошибки — `count < 1`, код ошибки хранится в `errno`.
       
- **Блокировка:** если системный вызов требует ожидания (например, чтение с клавиатуры), процесс блокируется, а ОС запускает другой процесс. Когда ресурс становится доступен, блокированный процесс продолжает выполнение.
   
- **POSIX и другие ОС:**
    
    - POSIX определяет набор процедур (~100), большинство из которых вызывают системные вызовы.
    - Mapping POSIX procedure → system call не всегда 1:1; иногда несколько библиотечных процедур используют один системный вызов.
    
- **Основные функции системных вызовов:**
    
    - Работа с процессами: создание, завершение.
    - Работа с файлами и каталогами: создание, удаление, чтение, запись.
    - Ввод/вывод с устройствами.

## System Calls for Process Management

**Основные системные вызовы:**

- `fork()` — создание дочернего процесса, точная копия родителя.
    - Возвращает **0** в дочернем, **PID** дочернего в родителе.
    - Использует **copy-on-write** для памяти: общий блок до изменения.
    
- `waitpid(pid, &status, options)` — ожидание завершения дочернего процесса.
    
- `execve(name, argv, envp)` — замена текущего процесса новым программным образом.
    
    - Используется для выполнения команд в дочернем процессе после `fork`.
    - `argv` — массив аргументов командной строки; `envp` — массив переменных окружения.
    
- `exit(status)` — завершение процесса, возврат статуса родителю через `waitpid`.


**Пример работы shell:**

1. Пользователь вводит команду.
2. Shell выполняет `fork()`, создавая дочерний процесс.
3. Дочерний процесс вызывает `execve()` для выполнения команды.
4. Родитель (`shell`) ждёт дочерний процесс через `waitpid()`.

**Структура памяти процесса:**

![800](Pasted%20image%2020251117210400.png)

- **Text segment** — код программы.
- **Data segment** — переменные программы (растёт вверх).
- **Stack segment** — стек вызовов (растёт вниз).
- Между стеком и данными — пустое пространство.

**Выделение памяти:**

- Для динамического расширения данных: `brk()` или, обычно, `malloc()`.
- Для создания новых виртуальных областей памяти: `mmap()`.

**Примечания:**

- `execve` может показаться сложным, но семантически это самый «тяжёлый» вызов POSIX; остальные проще.
- Процессы могут делить текстовую память, но данные и стек обычно отдельные.

## System Calls for File Management

**Основные системные вызовы для работы с файлами:**

- `open(name, flags)` — открыть файл.
    
    - `O_RDONLY`, `O_WRONLY`, `O_RDWR` — чтение, запись, или и то, и другое.
    - `O_CREAT` — создать новый файл.
    - Возвращает **file descriptor (fd)**.
    
- `close(fd)` — закрыть файл, освобождая дескриптор.
    
- `read(fd, buffer, nbytes)` — чтение `nbytes` байт из файла в буфер.
    
- `write(fd, buffer, nbytes)` — запись `nbytes` байт из буфера в файл.
    
- `lseek(fd, offset, whence)` — перемещает указатель текущей позиции файла:
    
    - `SEEK_SET` — от начала файла,
    - `SEEK_CUR` — от текущей позиции,
    - `SEEK_END` — от конца файла.
    - Возвращает абсолютную позицию после перемещения.
    
- `stat(name, &buf)` — получить информацию о файле: тип, размер, время последнего изменения и др.
    
- `fstat(fd, &buf)` — то же самое для уже открытого файла.


**Примечания:**

- Файлы могут читаться и писаться **последовательно** или **произвольно** (random access) с помощью `lseek`.
- Каждый файл отслеживает метаданные: режим, размер, время изменения и др.

## System Calls for Directory Management

**Основные вызовы для работы с каталогами и файловой системой:**

- `mkdir(name, mode)` — создать пустой каталог.
- `rmdir(name)` — удалить пустой каталог.
- `link(name1, name2)` — создать **жёсткую ссылку**: несколько имён (возможно в разных каталогах) ссылаются на один и тот же файл.
    
    - Важное: изменения через одно имя видны через другие.
    - У каждого файла есть уникальный **i-number** (inode). Ссылки используют один и тот же inode.
    
- `unlink(name)` — удалить имя файла. Если это последняя ссылка, освобождаются блоки на диске.
- `mount(special, name, flag)` — присоединить файловую систему (например, USB, SSD) к дереву каталогов.
    
    - Первый параметр — специальный файл устройства (`/dev/sdb0`).
    - Второй — точка монтирования в дереве (`/mnt`).
    - Третий — флаги (чтение/запись или только чтение).
    
- `umount(special)` — отключить файловую систему.

**Особенности:**

- Каталог — это файл с парами `(i-number, имя)`.
- Mount позволяет объединять разные устройства в одно **дерево каталогов**, скрывая физическое расположение.
- Поддерживается интеграция как постоянных, так и съемных носителей.

## Miscellaneous System Calls

**Основные вызовы:**

- `chdir(dirname)` — изменить текущий рабочий каталог.
    
    - Пример: `chdir("/usr/ast/test");` → открытие файла `xyz` будет обращаться к `/usr/ast/test/xyz`.
    - Упрощает работу с относительными путями.
    
- `chmod(name, mode)` — изменить права доступа к файлу (read/write/execute для владельца, группы и остальных).
    
    - Пример: `chmod("file", 0644);` → только владелец может писать, остальные читают.
    
- `kill(pid, signal)` — отправить сигнал процессу.
    
    - Процесс может перехватывать сигнал через **обработчик сигналов**.
    - Если сигнал не обрабатывается, процесс завершится.
    
- `time(&seconds)` — получить текущее время в секундах от 1 января 1970 года.
    
    - На 32-битных системах максимальное значение ~136 лет → проблема в 2106 году для 32-бит UNIX.

## The Windows API

Windows и UNIX отличаются в самой модели программирования:

• UNIX-программы работают как линейный код, который периодически вызывает системные вызовы.  
• Windows-программы **событийные**: основной поток ждёт события (клавиша, мышь, вставка USB) и вызывает обработчики.

Из-за этой модели Windows имеет иной подход к системным вызовам:

1. **WinAPI (Win32 / Win64)** — огромная библиотека функций, которые _не обязательно_ являются системными вызовами.
    
    - Некоторые функции идут в ядро.
    - Некоторые выполняются в user space.
    - И это может меняться от версии к версии.
    
2. WinAPI — стабильный программный интерфейс. Сами системные вызовы под ним Microsoft может менять, не ломая программы.
    
3. WinAPI гигантский: тысячи функций. Гораздо больше, чем POSIX (≈100).
    
4. Не всегда ясно, что именно является системным вызовом — это скрыто. В UNIX всё прозрачно: `read()` → системный вызов read.

---

### Соответствия UNIX ↔ Win32 (упрощённо)

• `fork + execve` → **CreateProcess**  
• `waitpid` → **WaitForSingleObject**  
• `exit` → **ExitProcess**  
• `open/close/read/write` → **CreateFile / CloseHandle / ReadFile / WriteFile**  
• `lseek` → **SetFilePointer**  
• `stat` → **GetFileAttributesEx**  
• `mkdir/rmdir` → **CreateDirectory / RemoveDirectory**  
• `chdir` → **SetCurrentDirectory**  
• `time` → **GetLocalTime**

Чего **нет** в Win32 (по классике):  
• сигналов,  
• ссылок (хардлинков),  
• mount/umount,  
• POSIX-стиля chmod.

(Но современные версии NT добавили часть функциональности — например, символические ссылки.)
